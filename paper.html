The Core Claim

One sentence.

We propose and implement the first tokenizer for Sanskrit that is compiled from grammar rules — not learned from data. Every token it produces is a real unit of meaning, derived from Pāṇini's 3,959-rule grammar system written around 400 BCE. We then benchmark it against today's standard tokenizers to show it works better for Sanskrit.

A tokenizer is the first thing an AI language model does: it breaks text into pieces called tokens before any learning happens. Today's tokenizers (BPE, SentencePiece) figure out which pieces to use by looking at billions of words and asking "what sequences of letters show up together most often?" They have no idea what words mean. They just find patterns.

Sanskrit does not need this. Pāṇini already wrote down the complete grammar — every valid Sanskrit word can be derived from around 2,000 roots using his rules. Deterministically. No data needed. The grammar is a compiler.

What's Already Verified

The foundation
is solid.

Every claim below is supported by independently verifiable published work. No AI-generated citations. All sources checked manually.

Sanskrit needs 2× fewer tokens than English for the same content

A January 2026 paper (arXiv 2601.06142) measured this directly. For dense Sanskrit commentary, the advantage can reach 20×. Each Sanskrit token carries more meaning than an English one — by design, not by accident.

Verified

Pāṇini's grammar is formally complete — and already runs as a computer program

Gérard Huet (INRIA, France) built the Heritage Platform: Pāṇini's 3,959 rules implemented as working software. Paul Kiparsky (Stanford) showed the grammar is formally more powerful than most modern grammar frameworks. The foundation exists. We did not have to build it.

Verified

The tools to build this exist today, for free

SanskritNLP (Python library), SanskritShala (IIT Kharagpur, published at ACL 2023), and the Heritage Platform are all available, open, and documented. This is an engineering project, not a research moonshot.

Verified

Two recent papers explicitly call for this — and neither builds it

Srikant et al. (IntelliSys 2025) argues for a Pāṇini-based model. arXiv 2601.06142 (Jan 2026) documents Sanskrit's token efficiency and calls for a grammar-aware tokenizer. Both papers identify the gap. Neither fills it. This proposal fills it.

Verified

Morpheme-based tokenizers already show LLM improvement (proxy evidence)

MorphPiece (2023) trained an LLM using morpheme-based tokens — not Pāṇini, but general morphology — and got comparable quality to standard GPT-2 at half the training cost. This is the strongest available proxy that the approach will work.

Verified
What We're Proposing to Build

The tokenizer.
Then the benchmark.

Step one: build a tokenizer that takes any Sanskrit text and splits it into morphemes — the smallest meaningful units — using Pāṇini's rules. Not by looking at frequency patterns. By applying grammar rules the way a Sanskrit scholar would.

Step two: run it against BPE and SentencePiece (the two standard tokenizers used in most AI models today) on the Digital Corpus of Sanskrit — 4.8 million words, freely available. Measure three things: vocabulary size, how accurately morpheme boundaries are found, and how many tokens are needed per sentence.

If the Pāṇinian tokenizer wins on morpheme boundary accuracy and token efficiency, the case is made. That is all the first paper needs to establish.

What's Still Being Tested

Honest about
what isn't proven yet.

The benchmark experiment has not been run yet

This is the most important missing piece. The experiment is straightforward — days, not months, using free tools on a free dataset. We are looking for a collaborator to run it.

Not yet run

Sanskrit word-splitting accuracy is 79.5% — not 100%

Sanskrit has sandhi: words fuse at boundaries in complex ways. Current tools split correctly about 4 out of 5 times. This is an engineering problem with a known improvement trajectory (ByT5-Sanskrit, TransLIST). It is not a hole in the theory — it is a solvable limitation.

Known limitation

Binary Matrix Glyphs are a future direction, not this paper

The BMG system — encoding each morpheme as a 2D binary matrix for machine-native reading — is proposed as Paper 2. The first paper makes no claims about it beyond naming it as a research direction. The tokenizer stands entirely on its own.

Verified References

What this is built on.

Every citation below was located, opened, and verified before being used. No AI-generated references.

  • arXiv 2601.06142 (January 2026) Documents Sanskrit's token efficiency advantage over English — 2× on average, up to 20× for dense commentary. Explicitly calls for a grammar-aware, Pāṇinian tokenizer. Does not build one. This proposal builds it.
  • Srikant et al., IntelliSys 2025 Argues for a Pāṇini-based AI model for Sanskrit. Makes the conceptual case without implementing the tokenizer. Confirms the gap exists in the literature.
  • Gérard Huet — Heritage Platform (INRIA, 2009) Pāṇini's grammar implemented as a working computational system. The morphological analyzer that makes this tokenizer possible already exists. We are applying it to ML pipelines, not rebuilding it.
  • Paul Kiparsky — "On the Architecture of Pāṇini's Grammar" (Springer, 2009) Formal analysis showing Pāṇini's grammar is at least as powerful as modern formal grammar frameworks — possibly Turing-equivalent. The theoretical foundation for the completeness claim.
  • MorphPiece (2023) Trained an LLM using morpheme-based tokens (not Pāṇini — general morphology). Achieved comparable quality to standard GPT-2 at half the training cost. The strongest proxy evidence that morpheme-aware tokenization improves LLM training.
  • SanskritShala — IIT Kharagpur (ACL 2023) Published Sanskrit NLP tools including word segmentation and morphological analysis. One of the tool sources for the proposed implementation.
